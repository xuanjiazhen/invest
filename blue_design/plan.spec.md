# 规划：内网需求 Issue 到开发完成的轻量方案

## 1. 交付件

| 文件 | 形式 | 内容 |
| --- | --- | --- |
| `req.spec.md` | 需求原文 | 保留原始输入，作为需求边界。 |
| `plan.spec.md` | 规划说明 | 明确本次只覆盖轻量链路，不展开完整社区治理。 |
| `report.spec.md` | 方案正文 | 输出目标设计、角色分工、平台职责、端到端链路、关键状态和风险控制。 |

## 2. 受众与目标

本次只设计一条轻量链路：内网需求按 SR/AR 粒度创建 Issue，使用 ACEHarness 集成的 AET 能力生成需求分析、架构设计和功能设计，设计文档进入 DesignDocs 仓库，DesignReview 平台负责全量设计刷新、版本设计审视、代码合入后一致性检查和 SEG 风险闭环。

不覆盖外部社区 Issue 治理、版本级需求评审、完整测试发布管理、跨 SIG 平台大看板和完整生命周期管理。

## 3. 内容结构

1. 目标态：说明轻量链路希望解决的问题和成功标准。
2. 对象模型：定义 SR、AR、Issue、版本设计资产、全量设计资产、不一致项。
3. 平台分工：划分 Issue、ACEHarness/AET、DesignDocs、DesignReview、代码仓和 SEG 的职责。
4. 端到端链路：从 Issue 创建到开发完成，按节点说明输入、动作、输出和责任人。
5. 文档与状态：定义设计资产目录、Issue 状态和不一致项状态。
6. 风险控制：说明低分设计、设计实现不一致、往期资产变化和测试补充的处理方式。

## 4. 审查准则

- 只围绕轻量链路展开，不引入全量社区开发活动。
- 所有新增平台职责必须能落到 DesignDocs 或 DesignReview 的具体能力。
- ACEHarness/AET 负责生成和辅助审查，不替代设计责任人和 SEG 风险决策。
- 设计资产必须能从 Issue、版本、代码仓、PR 和不一致项追溯。

## 5. 边界

本次输出是目标方案设计，不展开数据库模型、权限模型、平台接口、完整发布管理和前端页面。ACEHarness 评估基于本地仓库 `D:\docker\ace\ACEHarness`，AET 评估基于本地仓库 `D:\docker\aet`。
