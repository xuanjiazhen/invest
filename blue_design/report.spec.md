# 仓颉开源社区 AI 协同开发流程活动指导书

## 1. 目标与范围

本指导书面向仓颉开源社区的版本开发、Issue 治理、需求评审、设计评审、迭代开发、测试、发布和生命周期维护。目标是在 GitCode 社区协作基础上，引入 AI 辅助活动和可追溯交付件，让半年 STS 版本周期与月度迭代节奏具备统一入口、统一状态、统一证据链和统一看板。

适用范围包括三类输入：社区 forum 仓库中的外部反馈、具体特性仓库中的 Issue、版本规划初期导入的需求和问题。漏洞类事项按安全通道上报为主，仍需要进入版本级跟踪视图，并按评分约束纳入迭代入口评估。

## 2. 生命周期总览

半年 STS 版本周期分为洞察规划、需求与 Issue 收集、需求评审、分析及设计、迭代入口、迭代开发、测试与发布准备、发布与闭环八个阶段。版本初期完成整体洞察、需求导入和系统性设计；每个月度迭代开始前进行入口评估，确认最新社区 Issue、已规划内容、漏洞事项和跨 SIG 依赖。入口冻结后，范围变更进入下个迭代候选池。

## 5. Issue 生命周期与状态模型

Issue 进入社区后，先由 AI Robot 做规范审查、去重、分类和 Team 分派。合法 Issue 进入对应 SIG；信息不足或格式不合规的 Issue 标记为待补充，并给出修正建议。SIG 需要判断事项属于需求、缺陷、讨论或漏洞跟踪项，并给出答复。

需求类 Issue 在版本初期或月度入口前进入候选池，经需求评审后进入版本计划和迭代计划。缺陷类 Issue 经 Team 分析后进入候选池，非当月相关缺陷不强制冲击当前迭代，但必须在下个入口前完成分析。讨论类 Issue 以答复和共识闭环为主。漏洞类事项遵循安全上报通道，同时在版本看板中跟踪 SLA：评分大于 7 的漏洞目标为 30 天内完成版本发布，评分小于 7 的漏洞目标为 90 天内完成版本发布，适用于维护周期内版本。

推荐状态口径为：新建、AI 审查、待补充、Team 分析、候选池、评审通过、迭代已规划、设计中、入口冻结、开发中、待 QA 验证、迭代完成、待提出人确认、已关闭。等待提出人补充信息或确认解决结果时开始计时，超过一周无答复则默认关闭。无需 QA 验收的重构类或内部治理类事项，需要在迭代入口明确标记。

## 6. DesignSpec / SDD 交付件对齐

规划进迭代的需求类 Issue 需要提交特性设计包。设计包至少包含功能基本信息、功能规格与约束、功能设计说明书、DT 设计、架构影响、测试策略和风险分析。问题类 Issue 不要求完整设计包，但需要保留 Committer 审查、PR 关联、自测和回归证据。

![活动交付件与平台能力](./cangjie-community-ai-deliverables.svg)

源文件：[cangjie-community-ai-flow.drawio](./cangjie-community-ai-flow.drawio)

DesignSpec/SDD 对齐关系如下：

| 设计内容 | 建议承载 | 检查重点 |
| --- | --- | --- |
| 功能基本信息 | 功能定义信息 | 背景、目标、非目标、用户场景、范围边界 |
| 功能规格 | 功能规格和约束 | 触发场景、处理动作、输出结果、验收口径 |
| 功能约束 | 功能规格和约束 | 限制主题、边界条件、不满足时的处理结果 |
| 架构影响 | 功能设计说明书 / SDD | 模块关系、接口变化、数据流、兼容影响 |
| DT 设计 | 功能设计说明书 / SDD | 关键设计点、依赖、异常路径、质量属性 |
| 测试策略 | 功能设计说明书 / SDD、QA 测试设计 | 特性测试、SIT、回归范围、免测说明 |
| 发布信息 | Release Note、版本文档 | 用户可见变化、限制、迁移说明、文档验证 |
| 持续设计资产 | AI 刷新资产库 | 代码一致性、接口索引、架构视图、人工审视结论 |

设计资产分为两类：特性进入迭代时的提案型设计资产，用于记录需求分析和设计依据；代码合入触发刷新的持续型设计资产，用于保持代码视图、接口视图和架构视图一致。持续型资产由 AI 基于仓库内容生成和刷新，人工负责审视结论与例外处置。

## 7. 平台与看板能力

| 平台 | 现有定位 | 需要补齐的能力 |
| --- | --- | --- |
| GitCode Issue | 承载 forum、仓库 Issue 和社区讨论 | 跨仓 Issue 归一、模板差异适配、状态标签规范、等待提出人超时标记 |
| GitCode PR | 承载代码贡献、检视和合入 | PR 与 Issue 强关联、检视质量提示、自测证据要求 |
| opencsitool | 适合作为社区治理和迭代总看板 | 候选池、需求评审、版本计划、迭代入口冻结、漏洞 SLA、跨 SIG 依赖 |
| AI Robot / Agent | 自动审查、摘要和提醒 | Issue 合法性审查、Team 分派、评审包生成、设计质量评分、状态巡检 |
| DesignSpec/SDD 评审平台 | 承载设计模板和质量评审 | 功能定义、规格约束、设计说明书、DT、架构影响和测试策略的一体化追溯 |
| CI/CD 平台 | 构建、门禁、自测流水线 | 转测版本、回归版本、门禁策略、质量基线与 Issue 状态联动 |
| QA 测试管理 | 测试设计、执行、SIT 和回归 | DT 审视记录、串讲反串讲记录、回归结果与发布准出联动 |
| 文档发布平台 | Release Note 和版本文档发布 | Release Note 作为测试验收项，版本文档与 Issue/PR 追溯 |

opencsitool 的核心价值是把分散在 GitCode Issue、PR、CI/CD、QA 和文档平台中的状态聚合到版本和迭代视角。建议以 Issue 为主键，关联需求评审、设计包、PR、构建、测试、Release Note 和关闭记录。

## 8. 迭代准入与发布准出

迭代准入至少检查四类条件：关键特性设计已完成并通过评审；少量非关键特性的设计尾项有明确完成时间；漏洞 SLA 和高优缺陷有明确处置计划；QA、CI/CD、Doc SIG 的工作输入具备可执行性。准入结论应记录在迭代看板，冻结当月范围。

测试准入需要确认开发串讲完成、测试设计完成、转测版本构建成功、关键门禁通过、PR 与 Issue 关联完整。发布准出需要确认计划 Issue 的解决程度、遗留问题影响、SIT 结果、回归结果、Release Note 验证、版本文档和 PMC 审核结论。

## 9. 落地路线

第一阶段建立状态口径和最小看板：统一 Issue 类型、标签、等待提出人状态、迭代字段和 PR 关联要求。第二阶段补齐 AI Robot 能力：模板审查、分类分派、评审摘要、超时提醒、设计评分和 Release Note 校验。第三阶段打通证据链：以 Issue 为主键串联需求评审、DesignSpec/SDD、PR、构建、测试、发布和生命周期归档。

落地优先级建议从跨仓 Issue 归一和迭代入口看板开始，因为它们直接决定需求评审、范围冻结和后续证据链的可信度。AI 能力应嵌入各阶段入口和出口，不替代人工评审结论。
