# 规划：DesignSpec 外网迁移方案汇报材料

## 1. 交付件

| 阶段 | 产物 | 形式 |
|------|------|------|
| 需求 | `req.spec.md` | 已固化 |
| 规划 | `plan.spec.md` | 本文件 |
| 报告 | `report.spec.md` | 调研分析正文（现状 / 调整 / GAP 与计划） |
| 流程图 | drawio 绘制并导出 SVG | 内嵌进 slides，不单独索引 |
| 演示 | `DesignSpec外网迁移方案-slides.html` | 交互式网页 PPT |
| 索引 | `index.html` 新增卡片 | 仅索引 slides |

## 2. 受众与目标

- 受众：管理者 / 评审专家。
- 目标：在 10 页内讲清「现状 → 计划调整 → GAP 与落地路线」，让评审者能快速判断方案可行性与风险点。

## 3. 演示页面提纲（文字布局版，先审后做）

> 复用本仓库既有 slides 风格：1920×1080 舞台、红黑书法体标题、米色纸面、键盘/按钮翻页。

| 页 | 标题 | 版式 | 核心内容要点 |
|----|------|------|--------------|
| 01 | 封面：DesignSpec 设计评审平台外网迁移方案 | 封面（标题 + 副标题 + 标签行） | 主标题、一句话定位、汇报对象与日期 |
| 02 | 一、内网现状：能力与数据链路 | 左文右图（架构图） | DesignSpec 三层 MCP 取数链路（需求→SR→功能）+ AI 评审/收口能力 |
| 03 | 二、迁移目标与外网底座 | 三卡片 | opencsitool 提供版本+需求列表；gitcode wiki 承载功能设计；DesignSpec 改造取数分支 |
| 04 | 三、计划调整：数据来源重构（前后对比） | 双栏对照表 | 内网 MCP 三链 → 外网 opencsitool API + gitcode wiki 双源 |
| 05 | 四、目标态数据流（架构图） | 整页流程图 | opencsitool / gitcode wiki / DesignSpec 三方协作与关联关系 |
| 06 | 五、关联关系机制：需求 ↔ wiki | 左文右示意 | wiki 模板中标记 opencsitool 需求 ID，平台据此聚合取数 |
| 07 | 六、GAP 总览 | 矩阵表 | 按「数据接入 / 设计归档 / 关联一致性 / 评审闭环 / 安全合规 / 运维」列 GAP 与影响 |
| 08 | 七、关键风险与待决策项 | 双栏卡片 | 关联强约束缺失、wiki 质量参差、权限边界、辅助生成尚未落地等 |
| 09 | 八、落地路线（分阶段） | 阶段时间线 | 阶段一接入 opencsitool；阶段二接 wiki 取数；阶段三关联校验+评审；阶段四辅助生成 |
| 10 | 九、结论与决策建议 | 总结三卡片 | 可行性结论、需管理者拍板事项、下一步动作 |

## 4. 审查准则（生成前自检）

- 每页 ≤ 6 个要点，避免大段文字；标题统一「序号 + 短语」。
- 现状 / 调整 / GAP 三段式清晰可辨。
- 架构图四周留白、连线以直线为主，配色与 slides 一致。
- 不出现过程性叙述（合并 / 重画 / 原本 / 改为 等）。

## 5. 风险与边界

- DesignSpec 仓库本身无 opencsitool / gitcode / wiki 任何代码，迁移属新建数据接入层。
- 「辅助生成」当前平台尚未实现，仅评审 + 收口，路线中需单列。
