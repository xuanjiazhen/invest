# DesignSpec 设计评审平台外网迁移方案分析

## 摘要

将内网 DesignSpec 设计评审平台迁移至外网，核心是把数据来源从内网三层 MCP 取数链路，切换为外网 opencsitool（版本与需求列表）+ gitcode wiki（功能设计内容）双源，并通过「需求 ↔ wiki」关联关系把两类信息聚合给评审引擎。本文按现状、计划调整、GAP 与落地路线三段展开，供评审专家判断方案可行性与风险。

---

## 一、现状

### 1.1 平台定位

DesignSpec 是一套运行于内网的设计评审平台。输入功能名称 / 功能 ID（或整个项目版本），由本地 opencode AI 代理拉取功能定义与设计说明书，按评审规则打分，并将评审结果、改进建议与收口状态持久化在本地 SQLite。

仓库分为两层：

- 标准层：承载功能定义、功能设计模板、设计说明书规范与评审打分细则（功能定义 40 分 + 设计说明书 60 分）。
- 运行层：Node.js Web 应用，自动化执行评审、版本同步、收口检查与质量看板。

### 1.2 数据链路（内网）

平台自身不直连任何数据库或业务 API，全部取数由本地 opencode 代理调用三个 MCP 服务完成：

| 取数环节 | MCP 服务 | 返回内容 |
|----------|----------|----------|
| 版本 → SR 列表 | 需求获取 MCP | 版本下的 SR 标识 |
| SR → 功能列表 | SR 获取 MCP | 每个 SR 下的功能名 |
| 功能 → 详情 | 功能获取 MCP | 项目版本、功能名、功能定义 + 设计说明书 |

### 1.3 AI 评审与能力

- 评审：构造提示词 → opencode 调用功能获取 MCP 取数 → 按规则打分 → 输出结构化 JSON（分数、等级、维度拆分、P0/P1/P2 建议）→ 持久化。
- 收口检查：重新取数，逐条复核既有建议处于已收口 / 未收口 / 部分收口，可定时或手动触发。
- 版本同步：可断点续跑的分层流水线，基于内容摘要（SHA-256）做变更检测，仅对变更功能重评。
- 当前平台只做「评审 + 收口跟踪」，「辅助生成」能力尚未实现。

## 二、计划调整

### 2.1 总体思路

| 维度 | 内网现状 | 外网目标 |
|------|----------|----------|
| 运行环境 | 内网 | 外网可访问区域 |
| 版本与需求来源 | 需求 / SR MCP | opencsitool 平台（版本信息 + 需求列表） |
| 功能设计来源 | 功能获取 MCP | gitcode 仓库 wiki（模板 + 设计规范承载内容） |
| 需求与设计关联 | 系统内建链路 | wiki 内显式标记 opencsitool 需求关联关系 |
| 评审与生成 | 评审 + 收口 | 评审 + 收口 +（规划中）辅助生成 |

### 2.2 关键调整点

1. 新建 opencsitool 数据接入：取版本信息与需求列表，作为评审任务的入口与组织维度。
2. 开启 gitcode 仓库 wiki：下发功能设计模板与设计规范，作为功能设计内容的归档载体。
3. 建立「需求 ↔ wiki」关联：wiki 页面显式标记对应 opencsitool 需求 ID，平台据此把需求与设计内容聚合。
4. DesignSpec 改造取数分支：将三层 MCP 取数替换为「opencsitool API + gitcode wiki」双源取数，保留评审引擎与 JSON 契约不变。

### 2.3 关联关系机制

opencsitool 仅提供需求列表，不归档设计内容；gitcode wiki 承载设计内容。两者通过 wiki 模板中的需求关联字段绑定：平台先从 opencsitool 拉取需求清单，再按关联字段从 wiki 检索对应设计页，组装为「需求 + 设计」评审输入。关联字段的强约束与一致性校验是该机制成立的前提。

## 三、GAP 与落地路线

### 3.1 GAP 总览

| GAP 域 | 缺口描述 | 影响 |
|--------|----------|------|
| 数据接入 | DesignSpec 现无 opencsitool API 客户端与 gitcode wiki 取数能力 | 需新建外网数据接入层，替换 MCP 取数链路 |
| 设计归档 | opencsitool 无设计归档，依赖 wiki 模板规范化 | 模板未落地则设计内容结构不一致，影响打分维度提取 |
| 关联一致性 | 需求 ↔ wiki 关联靠人工标记，缺强约束 | 漏标 / 错标导致评审取数缺失或张冠李戴 |
| 评审闭环 | 收口状态依赖可重复取数与稳定标识 | wiki 页面改名 / 迁移会破坏标识，影响收口跟踪 |
| 安全合规 | 外网区域的访问权限、数据脱敏与凭据管理 | 内外网数据边界与权限模型需重新界定 |
| 辅助生成 | 平台尚未实现辅助生成 | 需求中的「辅助生成」属增量能力，需单列里程碑 |

### 3.2 关键风险与待决策项

- 关联强约束：是否在 wiki 模板中强制需求 ID 字段并做提交校验，避免人工漏标。
- wiki 质量参差：设计内容由人工撰写，结构与完整度不一，需规范 + 校验双手段托底。
- 权限边界：opencsitool 与 gitcode 的访问凭据、外网数据可见范围由谁审批与托管。
- 稳定标识：wiki 页面需以稳定 ID（而非标题）作为评审与收口的主键。
- 辅助生成范围：先做模板化草稿生成，还是直接做规范符合性生成，需管理者定边界。

### 3.3 落地路线（分阶段）

- 阶段一 · 接入 opencsitool：打通版本与需求列表取数，跑通评审任务入口与组织维度。
- 阶段二 · 接入 gitcode wiki：下发模板与规范，实现按需求关联检索设计内容。
- 阶段三 · 关联校验与评审闭环：建立关联强约束与一致性校验，恢复评审 + 收口全流程。
- 阶段四 · 辅助生成：在评审闭环稳定后，增量实现设计草稿辅助生成。

### 3.4 结论与建议

- 可行性：评审引擎与 JSON 契约可复用，迁移工作量集中在「外网数据接入层 + 关联机制」，方案在工程上可行。
- 需拍板：关联强约束策略、外网权限与凭据托管、辅助生成范围三项需管理者决策。
- 下一步：先落地阶段一接入与 wiki 模板规范，验证「需求 ↔ wiki」关联可用后再推进评审闭环。
