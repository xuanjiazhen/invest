一、背景与三种可选方案

仓颉作为鸿蒙第二语言，其开发指南、工具链指南、API 参考等资料部署有三个方向：

方案 A · 全量复制

所有资料从 cangjie-lang.cn 复制并适配至 developer.huawei.com，双站并存。

对标 Swift / Apple

方案 B · 仅语言站 + 链接

资料仅存 cangjie-lang.cn，鸿蒙侧只放简介，外链至语言官网。

对标 Kotlin / Android

方案 C · API 双站混合

标准库 API 双站部署，开发指南 & 工具链仅语言站 + 链接跳转。

折中方案

前提：标准库 API 参考已有结论——需放在 developer.huawei.com 与 ArkTS API 同级。本文聚焦非标准库部分。

四、竞品③：ArkTS on HarmonyOS

现状

  • ArkTS 无独立语言站（TypeScript 超集）
  • 全部资料统一在 developer.huawei.com
  • 指南 / API 参考 / 实践 / FAQ 与 HarmonyOS 文档树天然一体化
  • 开发者无需跨站，搜索一站式

启示

  • 鸿蒙开发者已习惯统一入口
  • 仓颉若分站部署，会产生"二等公民"感知
  • ArkTS 证明了统一平台站模式在华为生态可行
  • 但仓颉拥有独立语言站（cangjie-lang.cn），不能简单复制 ArkTS 模式

仓颉的挑战：既要像 ArkTS 一样融入鸿蒙开发者统一体验，又要保留独立语言站的合理存在价值。

八、推荐依据 & 落地建议

推荐依据

  1. Swift 是唯一可类比先例：同公司、有语言站、平台做主站
  2. Kotlin 模式不适用：分立源于跨公司组织边界
  3. 开发者期望：已习惯 ArkTS 统一入口
  4. API 已在平台站：指南跟随是自然延伸
  5. 语言站可转型：对标 swift.org 做开源社区

三阶段落地

阶段动作
短期developer.huawei.com 仓颉专区继续充实
中期cangjie-lang.cn 指南类加跳转提示，强化语言规范/工具链/社区
长期稳定分工：平台站 = 用仓颉开发鸿蒙，语言站 = 仓颉语言本身

九、结语

核心结论

  • 仓颉应走 Swift / Apple 路线
  • 开发者统一入口：developer.huawei.com
  • cangjie-lang.cn 转型为语言开源社区站

不推荐

  • Kotlin / Google 分站链接模式
  • 该模式反映的是跨公司组织边界
  • 仓颉 & 鸿蒙是同一家公司，不应制造人为割裂
