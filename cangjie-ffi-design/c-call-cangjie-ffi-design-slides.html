问题背景

现状：C 调用仓颉的痛点

当前只能通过 Cangjie.h 底层接口：

InitCJRuntime(¶m)          // 手工填写堆/GC/并发/日志参数
LoadCJLibraryWithInit("lib.so") // 加载动态库
FindCJSymbol("lib.so", "mangled_name") // 必须知道 mangle 后符号名
RunCJTask(fn, packArgs(...))    // 参数以 void* 打包
GetTaskRet(h, &ret)             // 获取返回值
  • 接口太过底层，使用门槛高
  • 与语言级 C 互操作机制（foreign/@C/CFunc/CType）完全割裂
  • 运行时加载过程全部暴露给用户

目标

  • 声明式导出：@C 标注顶层函数即可导出为 C 调用约定符号
  • 自动生成头文件：编译期产出 .h，类型映射自动完成
  • 放宽类型限制：允许 String、Array<T> 等仓颉原生类型
  • 隐藏运行时加载：初始化与协程调度封装在生成代码内部

关键发现：Swift 和 Go 都将运行时初始化隐藏在生成的包装代码中，用户只看到声明式导出 + 类型映射。

04 / 08

提案核心：@C 导出函数

基本语法

@C
func add(a: Int64, b: Int64): Int64 {
    return a + b
}

// 编译后自动生成头文件：int64_t add(int64_t a, int64_t b);

自定义符号名

@C["cj_math_add"]
func add(a: Int64, b: Int64): Int64 {
    return a + b
}

// 头文件：int64_t cj_math_add(int64_t a, int64_t b);

导出规则

  • 仅顶层函数可标记 @C
  • 参数和返回值类型必须满足扩展后的 CType 约束
  • 不支持下标的 CFunc 类型（简化为直接函数调用）
  • 生成代码自动处理协程调度（RunCJTask + GetTaskRet）

新增类型遵循"借用"语义：C 侧获得指针/视图在函数返回后失效，避免所有权混乱。

06 / 08

头文件自动生成 & 运行时封装

编译期头文件生成

  • 编译器扫描所有 @C 导出函数
  • 按类型映射表生成 C 声明
  • 自动包含 extern "C" 包装
  • 生成初始化/销毁入口函数：
    int cj_runtime_init(void);
    void cj_runtime_deinit(void);
  • 产出 {module}_export.h

运行时封装（C 侧透明）

  • 生成的 stub 函数内部自动：
    1. 调用 InitCJRuntime（首次）
    2. 加载仓颉动态库
    3. 参数类型转换（C → 仓颉）
    4. 通过协程调度执行目标函数
    5. 返回值类型转换（仓颉 → C）
  • C 调用方只需 #include "module_export.h" + 链接 .so
07 / 08

所有权与生命周期

借用语义（Borrow）

  • C 侧获得的指针/视图在函数返回后立即失效
  • 不转移所有权，C 侧无需释放
  • 适用于 String、Array<T> 等引用类型
  • 与 Swift @c 的 borrowing 语义对齐

初始化时序

  • 显式初始化：C 侧调用 cj_runtime_init()
  • 或 自动延迟初始化：首次调用任一导出函数时自动触发
  • 销毁：cj_runtime_deinit() 或程序退出时自动清理
  • 多线程安全：初始化互斥锁保护，后续调用无锁

源码兼容性 · 未来方向

方面内容
源码兼容新增功能不破坏现有 foreign/@C/CFunc 机制；Cangjie.h 底层接口保留但不再推荐直接使用
未来方向支持异步导出函数（@C async）、回调函数类型、更丰富的集合类型映射（Map/Set）

总结

提案核心价值

  • 声明式：@C 标注即可，与仓颉→C 方向一致
  • 自动化：编译期生成头文件 + 运行时 stub
  • 透明：C 侧无需了解仓颉运行时细节
  • 对标业界：Swift @c 和 Go cgo 的成熟模式

待决策项

  • 字符串编码策略（UTF-8 / UTF-16？）
  • Array 视图的内存布局约定
  • 错误处理（异常如何跨 FFI 边界？）
  • 异步导出函数的协程模型
