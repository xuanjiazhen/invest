ABI：@C 与 repr(C) 都不是头文件证明

仓颉

CType 限定可进入 FFI 的类型；@C struct 约束 C 兼容布局。

Rust

#[repr(C)] 固定 Rust 类型的 C 布局。

共同边界：都不会证明本语言声明与某个 C 头文件一致。宽度、packed、位域、联合体、平台 ABI 与头文件漂移仍需生成器和 layout tests。

04 / 12

CPointer：能判空，但不携带契约

已有能力

let p = CPointer<Int32>()
if (p.isNull()) {
    // 显式处理 null
}

缺失语义

长度、生命周期、所有权、分配器、线程约束均不在 CPointer<T> 中。

正确表述不是“仓颉没有 Option 所以无法判空”，而是 raw ABI 无法强制检查；Option 适合安全包装层。

05 / 12

堆数组：真实 acquire/release API

let handle = acquireArrayRawData(values)
try {
    use_buffer(handle.pointer, values.size)
} finally {
    releaseArrayRawData(handle)
}

API 语义

返回 handle；传 C 的地址是 handle.pointer；两端调用均处于 unsafe。

必须保证

C 不越界、不 free、不在 release 后保存或使用地址；长度单位一致；所有路径成对 release。

06 / 12

inout：调用期间有效的地址

foreign func write_value(p: CPointer<Int32>): Unit
var value: Int32 = 0
unsafe { write_value(inout value) }

inout 是调用点修饰符，只能用于 CFunc 调用。字面量、let、入参、临时值及源自 class 实例成员的值受到限制。

关键契约

地址仅在函数调用期间保证有效；C 不应保存。编译器约束可传对象，但无法审计任意 C 是否偷存指针。

foreign func write_value(p: CPointer<Int32>): Unit
foreign func sum_values(p: CPointer<Int32>, n: Int64): Int64
main(): Int64 {
 var value: Int32 = 0
 var values = [Int32(10), Int32(20), Int32(30)]
 unsafe {
  write_value(inout value)
  let handle = acquireArrayRawData(values)
  try { if (sum_values(handle.pointer, values.size) != 60) { return 3 } }
  finally { releaseArrayRawData(handle) }
 }
 if (value == 42) { 0 } else { 4 }
}

10 / 12

堆与栈：差异在约束强度，不在口号

仓颉

数组 acquire/release 固定借用；inout 明确只在调用期间有效；二者均需 unsafe。

Swift / Rust / Go

Swift withUnsafe* 形成词法作用域；Rust 安全借用转 raw 后进入 FFI 契约；Go cgo 有指针保留规则和运行时约束。

共同限制：C 仍可能越界、错误释放或保存借用地址。unsafe/作用域/运行时检查只降低误用，不把 C 变成内存安全语言。

绑定生成与防护工具

仓颉现状

官方 HLE cj-c2cj 可从 C 头文件生成胶水；社区 cjbind 提供 C/C++ 包装路线。

检测

GWP-Asan 在 Linux/OpenHarmony 抽样检测 acquire 数组的部分越界写与未 release；非全量且 Windows 不支持。

真正差距：覆盖面、稳定性、所有权元数据、布局回归和 CI 生态成熟度，而不是“仓颉完全没有自动工具”。

12 / 12

建议：把契约变成可执行护栏

  1. 生成而非手写：头文件导入 + 声明漂移检测 + layout tests。
  2. raw 层私有化：对外暴露 Resource、Option、复制型字符串/数组、pointer+len。
  3. 作用域借用：封装 withArrayRawData，在 finally 自动 release。
  4. 所有权元数据：记录 borrowed/owned、nullable、length-of、allocator。
  5. 真实 E2E：C DLL 编译、cjc 链接运行、ASan/GWP-Asan、异常路径与模糊测试。

结论：仓颉已有重要防线；核心机会是让布局、长度、生命周期和释放契约可生成、可验证、可持续回归。
