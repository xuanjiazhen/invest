风险 1：ABI 与类型映射失配

合法声明，不等于正确声明

典型入口

  • int64_t 被声明为 Int32
  • size_t / long 平台宽度理解错误
  • packed、位域、联合体、柔性数组无法等价表达
  • 头文件升级，手写 foreign 声明滞后
  • CDECL / STDCALL 双方不一致
可能后果参数误读返回值截断数据损坏 / 崩溃
CType 与 @C 证明“可用于 C ABI”；生成绑定与 layout tests 才能核对“是否匹配这个 C API”。

风险 2：空指针、悬垂与类型混淆

CPointer 能判空，但不携带生命周期
let p = unsafe { lookup() }
if (p.isNull()) {
    return None
}
let value = unsafe { p.read() }

判空之后仍可能出错

  • C 已释放对象，地址仍为非 null
  • 返回的是内部短期缓冲区
  • CPointer<A> 被转换为错误的 B
  • 偏移落到对象范围之外

Option 适合安全包装层；原始 ABI 仍以可为 null 的 CPointer 表达，无法强制每个调用方检查。

风险 4：长度、边界与整数换算

裸指针没有“剩余多少元素”
pointer + length + capacity + unit

常见失配

  • 元素数与字节数混淆
  • 负数转无符号后成为超大值
  • Int64 → Int32 截断

写入型 API

  • 输出长度超过容量
  • length × sizeof(T) 溢出
  • 只返回指针，未提供可信长度

固定数组可用 VArray；动态缓冲区应把地址、长度与容量作为不可拆分的包装。

风险 5：仓颉堆数组借给 C

固定借用解决移动问题，不解决越界与逃逸
let handle = acquireArrayRawData(values)
try {
    consume(handle.pointer, values.size)
} finally {
    releaseArrayRawData(handle)
}

借用期内的五条底线

  1. C 不越界
  2. C 不 free
  3. C 不保存地址
  4. 所有路径都 release
  5. 并发访问有同步
GWP-Asan 可抽样发现部分越界写和未 release；它不是全量证明。

风险 6：inout 地址逃逸

调用期有效，不代表 C 无法偷存
仓颉局部值var value: Int32
调用期间write_value(inout value)
调用返回后C 再次访问 → 悬垂

仓颉限制

必须可变且满足 CType；不能是 let、字面量、入参、临时表达式或 class 实例成员来源。

剩余责任

C 不得写入全局变量、异步任务或延迟回调上下文。

风险 8：回调、线程与错误边界

地址之外，还有执行上下文的生命周期

回调

注销后继续调用、userdata 已释放、销毁与回调并发

线程

C TLS、线程亲和性、长时间阻塞与仓颉线程调度冲突

错误码、errno、部分初始化、longjmp 导致清理遗漏

包装层需要定义注册期、线程、重入、阻塞和错误转换，而不只是声明函数签名。

void write_value(int32_t *p) {
    if (p) *p = 42;
}
int64_t sum_values(const int32_t *p, int64_t n) {
    int64_t s = 0;
    for (int64_t i=0; i<n; ++i) s += p[i];
    return s;
}

write_value(inout value)

最终判断

风险核心是契约没有完整进入类型与工具链
仓颉已经能标记危险边界，也能约束部分类型、调用和借用行为；但 C ABI、长度、所有权、释放器和有效期仍分散在头文件、文档与人工约定中。
P0
生成绑定 + 布局核验P0
所有权与长度元数据P1
作用域 API + ResourceP2
检测与持续 E2E

目标不是让 raw pointer 看起来更安全，而是让每一条跨边界契约可表达、可生成、可验证。
