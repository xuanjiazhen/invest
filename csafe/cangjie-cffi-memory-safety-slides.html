仓颉 C-FFI 两大机制

方向一：C → 仓颉（声明式调用）

// foreign 声明导入 C 符号
foreign func printf(fmt: CString, ...): Int32

// @C struct 共享内存布局
@C struct Point { var x: Float64; var y: Float64 }

// CFunc 表示 C 函数指针
var cb: CFunc<(Int32) -> Unit>

方向二：仓颉 → C（运行时桥接）

// Cangjie.h 底层接口（现状）
InitCJRuntime(&param)           // 手工配置堆/GC/并发参数
LoadCJLibraryWithInit("lib.so") // 加载动态库
FindCJSymbol("lib.so", "mangled") // 查找符号
RunCJTask(fn, packArgs(...))    // 协程调度执行

// @C 导出（提案 CJ-0001，未落地）
@C public func add(a: Int64, b: Int64): Int64

核心风险：两套机制共享同一套类型系统和内存模型，但编译器不提供 FFI 边界的安全检查。

问题 ① ABI 不匹配：类型映射错误

整型宽度不匹配

// C 侧：int64_t add(int64_t a, int64_t b);
foreign func add(a: Int32, b: Int32): Int32
// ❌ ABI 不匹配：C 侧 8 字节，仓颉侧 4 字节
// 后果：参数错位、返回值截断、栈帧破坏

结构体对齐差异

@C struct S {
    var a: Int8   // offset 0
    var b: Int64  // offset 8（自然对齐）
}
// C 侧 #pragma pack(1) → b 在 offset 1
// 跨平台 32/64 位也不同

CFunc 调用约定不匹配

var cb: CFunc<(Int32) -> Unit>  // 默认 cdecl
// C 侧 typedef void (__stdcall *cb)(int);
// ❌ 调用约定不匹配 → Windows 上栈失衡

CString 编码假设

foreign func getName(): CString
let s = String(getName())
// 默认 UTF-8，若 C 侧返回 GBK/Latin-1
// → 乱码或 UTF-8 解码失败 crash

问题 ② 指针安全：悬垂 & 双重释放

Use-after-free

foreign func createData(): CPointer<Int32>
foreign func freeData(p: CPointer<Int32>): Unit

let p = createData()
freeData(p)          // C 侧 free
let v = p[0]         // ❌ use-after-free
// CPointer 无生命周期，编译器无感知

Double-free

let p: CPointer<UInt8> = malloc(100)
free(p)               // 仓颉侧释放
// ... 跨 FFI 返回给 C 侧 ...
// C 侧再次 free(p) → ❌ double-free
// 无所有权转移，两侧各自释放

空指针：无 Option 语义

foreign func mayReturnNull(): CPointer<Int64>
let p = mayReturnNull()  // 可能为 NULL
let v = p[0]               // ❌ 段错误
// CPointer 无 Option<CPointer<T>> 惯用表达

数组越界：无长度信息

foreign func getArray(): CPointer<Int32>
let p = getArray()
for (i in 0..100) {        // 不知道实际长度
    process(p[i])          // ❌ 越界读写
}
// CPointer 不携带长度信息

问题 ② 指针安全：越界 & 类型混淆

指针算术无边界检查

let p: CPointer<UInt8> = getBuffer()
let offset: Int64 = getOffset()
let q = p + offset   // 无任何边界校验
// offset 为负数 → 读任意前置内存
// offset 超界 → 读任意后置内存
// 攻击者可实现任意地址读写

类型混淆 / Reinterpret Cast

// C 侧返回 void*
foreign func getHandle(): CPointer<Unit>

let raw = getHandle()
// 仓颉侧 reinterpret 为任意类型
let asFloat: CPointer<Float64> = unsafe { raw }
asFloat[0] = 1.0  // ❌ 若实际是 Int32 数组

多线程竞态

场景风险
C 线程 free + 仓颉协程读use-after-free 竞态
C 线程写 + 仓颉协程写数据竞争、未定义行为
GC 线程回收 + C 线程持有指针GC 回收后 C 侧悬垂

CPointer 无 Sync/Send trait，多线程场景无编译期保护。

问题 ③ 仓颉堆内存暴露到 C 侧

acquireRawData：String 原始指针

let s = "Hello, 仓颉"
let raw = acquireRawData(s)  // CPointer<UInt8>
// raw 指向 GC 托管堆上的 String 内部缓冲区
// 风险：
// 1. GC 回收 s → raw 悬垂
// 2. C 侧 free(raw) → 破坏 GC 堆
// 3. C 侧写入 raw → 破坏 String 不可变性

acquireArrayRawData：Array 缓冲区

var arr = [1, 2, 3, 4, 5]
let raw = acquireArrayRawData(arr)
// 危险 1：arr 不再引用 → GC 回收
// 危险 2：arr.append(6) → realloc
//   旧缓冲区失效，raw 悬垂
// 该指针是瞬时的，C 侧必须用完即弃

acquire/release 配对遗漏

let p: CPointer<UInt8> = acquire(ptr)
// ... 传给 C 侧使用 ...
if (someCondition) { return }  // ❌ 提前返回
release(ptr)  // 永远执行不到 → 内存泄漏
// 无 RAII，依赖程序员纪律

C 侧长期持有堆指针

func onData(cb: CFunc<(CPointer<UInt8>)>) {
    let buf = acquireArrayRawData(globalBuf)
    cb(buf)   // 传给 C 侧回调
} // 函数返回 → buf 可能失效
// 若 C 侧将 buf 保存为全局变量 → 💥

问题 ④ 仓颉栈内存暴露到 C 侧

inout 参数暴露栈地址

foreign func readBuf(buf: CPointer<Int32>, len: Int32)

func processData() {
    var buf = Array<Int32>(100, item: 0)
    var raw = acquireArrayRawData(buf)
    readBuf(raw, 100)  // C 侧写入
} // buf 在栈上？在堆上？取决于编译器
// 若分配在栈上 → C 侧获得栈地址
// 函数返回后 C 侧继续用 → use-after-return

栈缓冲区溢出

foreign func strcpy(dst: CPointer<UInt8>, src: CString)

func vulnerable() {
    var buf = Array<UInt8>(16, item: 0)
    let raw = acquireArrayRawData(buf)
    let long = "overflow string..."
    strcpy(raw, long)  // ❌ 溢出 16 字节
}
// CPointer 无边界保护 → 栈破坏/代码执行

竞品关键安全设计借鉴

Swift：withUnsafe 闭包模式

对堆/栈内存的所有 FFI 暴露都限定在闭包内

// Swift 闭包保证指针生命周期
var arr = [1, 2, 3]
arr.withUnsafeBufferPointer { buf in
    c_function(buf.baseAddress, buf.count)
} // 闭包结束 → 指针立即失效

仓颉可借鉴：用 withRawData 闭包替代 acquireRawData。

Rust：unsafe 块 + borrow checker

FFI 调用必须包裹在 unsafe 块中

// Rust unsafe 块声明危险操作
let slice: &[i32] = &vec;
unsafe {
    c_function(slice.as_ptr(), slice.len());
} // borrow checker 保证 vec 存活

仓颉可借鉴：引入 unsafe 上下文 + CBuffer<T> 携带长度。

Go cgo：指针 pinning + GoSlice

cgo 在调用期间 pin 住 Go 指针，GoSlice 携带长度

// Go cgo 自动 pin 指针
// C.CBytes 需手动 free
s := "hello"
cs := C.CString(s)        // 复制到 C 堆
defer C.free(unsafe.Pointer(cs)) // defer 保证释放

综合对比结论

模式来源仓颉适配
闭包生命周期Swift需新增
unsafe 标注Rust需新增
指针 pinningGo需新增
携带长度/容量Go/Rust/Swift需新增 CBuffer

总结与建议

态势判断

  • "信任程序员"模型：所有安全检查依赖手动保证
  • acquire 系列是最大风险点：GC 堆指针无生命周期追踪
  • CPointer<T> 是 C 语言 T* 的薄封装：无长度、无 Option、无所有权
  • 与竞品差距显著：Swift/Rust/Go 均有系统性防护
  • CJ-0001 提案方向正确：但未解决 CPointer/acquire 根本问题

优先投入

  • P0：acquire 系列改为闭包形式（withRawData）
  • P0：引入 CBuffer<T>，携带 data + len
  • P1：CPointer<T> 支持 Option 语义
  • P1：编译期对齐/宽度校验
  • P2：引入 unsafe 上下文标注
  • P2：CPointer 生命周期标注
