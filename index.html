Research Report / HarmonyOS Static Checks

鸿蒙上 ArkTS 与仓颉编译期静态检查能力对比

覆盖代码坏味道、类型与规范、安全、性能调优、多端适配、命令行集成、告警屏蔽等开发期检查能力。结论基于 2026-06-05 可访问的官方文档与仓颉公开文档取证。

报告形式：PPT 式网页 ArkTS：Code Linter + codelinter + HiChecker 仓颉：cjlint + 编译器语义检查 仓颉华为页：需授权账号
Executive Summary

总览结论

8

ArkTS 规则集族

通用 TypeScript、ArkTS 风格、安全、性能、预览、一次开发多端部署、正确性、兼容性。

40+

仓颉 cjlint 默认规则

覆盖命名、格式、声明、函数、接口、操作符、表达式、错误处理、并发、安全、序列化等。

CI

两侧均可工程化

ArkTS codelinter 支持退出码、输出格式、QuickFix；仓颉 cjlint 支持 JSON/CSV、规则/文件/注释屏蔽。

判断：ArkTS 的优势在于与 HarmonyOS/ArkUI/多端体验深绑定，尤其是性能与兼容性规则；仓颉的优势在于语言规范类、安全类、并发类规则更集中，且可通过禁用语法做架构约束。两者不是简单替代关系，更像“应用框架体验检查”与“通用语言规范/安全检查”的侧重差异。

Toolchain

工具形态对比

ArkTS

以 DevEco Studio 的 Code Linter 为核心，面向 TS/ArkTS 最佳实践、ArkUI 性能、多端体验和 API 兼容性。支持 Full Linter、Incremental Linter、QuickFix、Excel 导出，并提供 codelinter 命令行用于门禁和 CI。

code-linter.json5 ruleSet / rules / overrides --fix --exit-on json / xml / html

仓颉

以 CJLint 为核心，基于仓颉语言编程规范识别不符合 Clean Source 的问题。工具通过模块编译方式分析源码目录，支持规则级、源码注释、文件级告警屏蔽，并可按 JSON/CSV 输出报告。

cjlint -f src cjlint_rule_list.json exclude_lists.json cjlint-ignore json / csv
Code Smells

命名规范@typescript-eslint/naming-convention、@hw-stylistic/file-naming-conventionG.NAM.01~05 包名、源文件名、类型名、函数名、常量名。
格式风格@hw-stylistic/* 覆盖缩进、括号、空格、引号、行宽、分号间距。G.FMT.01 UTF-8，部分版本还有浮点数字面量格式等。

结论：常规坏味道两者都具备。ArkTS 更像 ESLint 生态扩展，配置弹性高；仓颉规则命名更接近语言规范条款，默认清单结构更清晰。
Type Safety

类型安全与接口边界

ArkTS

重点围绕 TypeScript/ArkTS 的类型边界收紧：禁止显式 any、禁止不安全的 any 参数/赋值/调用/成员访问/返回，要求函数、导出函数、公共类方法显式声明返回类型和参数类型。

@typescript-eslint/no-explicit-any @typescript-eslint/no-unsafe-call @typescript-eslint/explicit-function-return-type @typescript-eslint/explicit-module-boundary-types

仓颉

仓颉本身是静态强类型语言，编译期类型检查是基础能力；cjlint 在规范层补充 public 变量类型和 public 函数返回类型显式声明、接口使用方式、泛型约束接口使用等规则。

G.DCL.02 G.ITF.01 G.ITF.03 G.ITF.04
Performance

性能与调优类检查

ArkTS 的性能规则明显更贴近 HarmonyOS 应用真实场景，尤其是 ArkUI 渲染、组件复用、状态变量访问、动画、图片、Web、AVPlayer、DataShare 资源释放等。

@performance/high-frequency-log-check @performance/hp-arkui-use-reusable-component @performance/hp-arkui-no-state-var-access-in-loop @performance/hp-arkui-use-object-link-to-replace-prop @performance/bad-deep-clone-check @performance/reuse-date-instances-check @performance/datashare-query-unrelease-check
仓颉 cjlint 暂未看到同粒度的 UI/应用性能规则，但通过 G.VAR.01 优先不可变变量、G.VAR.02 缩小变量作用域、G.EXP.* 控制副作用、并发规则避免死锁/数据竞争，可以间接提升可维护性与运行质量。
Security

安全编码检查

安全主题ArkTS仓颉
加密算法安全禁止不安全 AES ECB、DH/DSA/RSA/3DES、弱哈希和 MAC。公开 cjlint 清单未以算法规则为主。
输入校验可通过自定义规则补充业务限制。G.CHK.01 跨信任边界不可信数据使用前必须校验。
日志与敏感信息安全规则包含注释代码清理，性能规则限制高频 Hilog。G.CHK.02、G.OTH.01 禁止外部数据/敏感信息直接进日志。
硬编码与敏感数据存储可通过自定义安全规则扩展。G.OTH.02 禁止硬编码敏感信息；G.OTH.04 不用 String 存储敏感数据。
序列化安全未在已取证规则中看到专门族。G.SER.01~03 覆盖敏感数据、构造方法绕过、类型一致性。
Concurrency & Resources

并发、锁与资源泄漏

ArkTS

静态侧重点在应用性能和资源使用，如 DataShare 查询结果及时关闭。运行期开发检测方面，HiChecker 可检测耗时函数调用、Ability 连接泄漏、ArkUI 性能，并通过日志或触发崩溃反馈。

RULE_THREAD_CHECK_SLOW_PROCESS RULE_CHECK_ABILITY_CONNECTION_LEAK RULE_CHECK_ARKUI_PERFORMANCE @performance/datashare-query-unrelease-check

仓颉

cjlint 对并发安全有直接规则：禁止暴露系统内部锁对象、异常路径释放锁、禁止用非线程安全函数覆写线程安全函数、按相同顺序请求锁避免死锁、避免数据竞争。

G.CON.01 G.CON.02 G.CON.03 P.01 P.02
HarmonyOS-Specific

正确性、兼容性与多端体验

这是 ArkTS Code Linter 的明显优势区：规则直接面向 HarmonyOS API、ArkUI 状态模型、媒体、网络、多设备布局与可访问性。

正确性

AVSession 按钮/元数据、音频中断、图像格式、网络变化监听、冗余依赖、V1 状态对象更新格式等。

兼容性

颜色对比度、颜色资源引用、字体单位与大小、栅格、触摸热区、断点判断、沉浸式效果、窗口尺寸监听。

仓颉公开 cjlint 文档未看到 HarmonyOS UI/多端体验规则；若授权文档中有 IDE 集成规则，应在后续补证后更新。
Architecture Guardrails

禁用语法与架构约束

ArkTS

通过 @typescript-eslint/no-restricted-syntax 可以用 AST selector 禁用特定语法或调用，例如禁止调用 Foo.bar()。优势是表达能力灵活，适合项目级规范。

仓颉

G.SYN.01 提供禁用语法检查，可配置 Import、Let、Spawn、Synchronized、Main、MacroQuote、Foreign、While、Extend、Operator、Class、Generic、Match、TryCatch 等关键字。

判断：仓颉在“统一限制语言子集”上更开箱即用；ArkTS 更适合基于 ESLint AST DSL 做细粒度项目约束。
